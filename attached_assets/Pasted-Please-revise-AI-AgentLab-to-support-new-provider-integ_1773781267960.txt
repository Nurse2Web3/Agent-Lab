Please revise AI AgentLab to support new provider integrations, update pricing-tier access, and visibly list provider availability throughout the website and app.

PROJECT GOAL
AI AgentLab should feel like a founder-friendly AI comparison studio for real builders. The core value is running the same prompt across multiple AI providers and comparing outputs side by side. The product should feel premium, useful, modern, and decision-oriented. Do not describe it as a playground.

CORE PRODUCT EXPERIENCE
Users should be able to:
- enter one prompt
- send that same prompt to multiple AI providers
- view outputs side by side
- compare response speed
- compare token usage when available
- compare estimated cost when available

PROVIDER INTEGRATIONS TO SUPPORT
Please structure the codebase to support these providers:

- Google Gemini
- Grok
- Kimi
- OpenAI
- Claude

IMPORTANT IMPLEMENTATION REQUIREMENTS
- Build provider integrations using a provider abstraction layer so each provider has its own adapter/service file
- Keep API keys in environment variables only
- Do not hard-code API keys
- Normalize provider responses into one common format for display in the comparison UI
- Normalize these fields when available:
  - provider name
  - model name
  - output text
  - latency
  - token usage
  - estimated cost
  - error state
- Make the architecture easy to expand later with more providers

SUGGESTED ENVIRONMENT VARIABLES
- GEMINI_API_KEY
- GROK_API_KEY
- KIMI_API_KEY
- OPENAI_API_KEY
- ANTHROPIC_API_KEY

TECHNICAL BUILD DIRECTION

Please organize integrations so they are production-friendly.

Recommended structure:
- /lib/providers/gemini.ts
- /lib/providers/grok.ts
- /lib/providers/kimi.ts
- /lib/providers/openai.ts
- /lib/providers/claude.ts
- /lib/providers/types.ts
- /lib/providers/router.ts

Requirements:
- each provider adapter should accept a prompt and return a normalized response
- the comparison runner should call only the providers allowed by the user’s plan
- if one provider fails, the UI should still show the other provider responses and display a clean error for the failed one
- add placeholder comments where actual API request code needs to be inserted if any provider is not yet fully wired

MODEL CONFIGURATION REQUIREMENT
Do not hard-code old or fragile model names across the UI.
Create a central provider config object for:
- provider label
- default model
- fallback model
- plan availability
- active/inactive status

This is important because provider model availability can change.
